## Reference
- Zitadel mgmt API: https://zitadel.com/docs/apis/introduction
- `/management/v1/orgs/_search` — listar orgs
- `/management/v1/users/_search` — listar users
- `/management/v1/projects/_search` — listar projects
- `/management/v1/projects/{projectId}/apps/_search` — listar OAuth clients
- `/admin/v1/policies/...` — policies system-wide
- `/system/v1/instances/me` — metadata de la instancia (version)

## Prompt
1. Cargar PAT a var: `PAT=$(bash $HOME/.claude/skills/mi-key-cli/scripts/mkey.sh pull bitacora prod --stdout | grep -E '^ZITADEL_ADMIN_PAT=' | cut -d= -f2-)`. Validar: `[[ -n "$PAT" ]]` o STOP.
2. **Version Zitadel:** `curl -sS -H "Authorization: Bearer $PAT" https://id.nuestrascuentitas.com/system/v1/instances/me | jq '.instance.version'` (puede requerir scope system — si 403, fallback: `curl -sS https://id.nuestrascuentitas.com/debug/healthz` + inspeccion de image tag de T1.1).
3. **Orgs:** `curl -sS -X POST -H "Authorization: Bearer $PAT" -H "Content-Type: application/json" -d '{"query":{"offset":"0","limit":100}}' https://id.nuestrascuentitas.com/management/v1/orgs/_search > /tmp/orgs.json`. Extraer `[{id, name, state, primary_domain}]`. Verificar si las 4 requeridas existen: `nuestrascuentitas`, `bitacora`, `multi-tedi`, `gastos`. Flaggear cuales faltan — esas son el input de T2.1.
4. **Users admin:** `curl -sS -X POST -H "Authorization: Bearer $PAT" -H "Content-Type: application/json" -d '{"query":{"offset":"0","limit":100}}' https://id.nuestrascuentitas.com/management/v1/users/_search > /tmp/users.json`. Filtrar users con `type=human` y username que termine en admin o similar. Reportar: cantidad total, email del admin actual, si tiene MFA enrollada (`/management/v1/users/{userId}/auth-factors/_search`).
5. **Projects + OAuth Clients:** `curl ... /management/v1/projects/_search` -> IDs. Por cada project, `curl ... /management/v1/projects/{projectId}/apps/_search` -> lista de apps (OAuth clients, SAML, APIs). Reportar: `[{projectId, projectName, apps: [{appId, name, oidcConfig: {clientId, redirectUris, responseTypes, grantTypes}}]}]`. Verificar si existen clients para bitacora/nuestrascuentitas/multi-tedi/gastos (fuente de T2.4).
6. **SMTP config:** `curl -sS -H "Authorization: Bearer $PAT" https://id.nuestrascuentitas.com/admin/v1/smtp > /tmp/smtp.json`. Reportar: host, port, sender, tls enabled. Si `404` o vacio, flaggear "SMTP NO wireado" — eso es input de T2.2.
7. **Policies admin:**
   - Password complexity: `/admin/v1/policies/password/complexity`
   - Password age: `/admin/v1/policies/password/age`
   - Lockout: `/admin/v1/policies/lockout`
   - Login (MFA forzada?): `/admin/v1/policies/login`
   - Privacy: `/admin/v1/policies/privacy`
   - Audit log retention: `/admin/v1/policies/audit_log`
   Reportar cada una. Marcar cuales hay que endurecer en T2.5.
8. **Audit log sample:** `curl ... /management/v1/events/_search -d '{"limit":10}'` — pedir ultimos 10 eventos. Util para verificar que auditoria esta activa.
9. Persistir el reporte con la estructura del skeleton abajo.

## Execution Procedure
1. Load PAT.
2. Ejecutar los 7 bloques de curl. Capturar response JSON y pretty-print con `jq`.
3. Cross-check con T1.1 que versiones concuerden.
4. Si cualquier curl devuelve 401/403: PAT no tiene scope suficiente -> parar y escalar al user para regenerar PAT.
5. Persistir reporte + commit.
