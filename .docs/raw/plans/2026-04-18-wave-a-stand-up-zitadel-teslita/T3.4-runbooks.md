# Zitadel Teslita — Bootstrap Runbook

> Cuando usar: primer deploy o recreacion total de `id.nuestrascuentitas.com`.
> NO usar para parches incrementales — para eso ver zitadel-recovery.md.

## Pre-requisitos
- DNS `id.nuestrascuentitas.com` A-record apunta a VPS (`54.37.157.93`), sin proxy Cloudflare
- DOKPLOY_API_KEY con scope full en Infisical
- ZITADEL_MASTERKEY nuevo de 32 bytes (generar: `openssl rand -base64 32`)
- Cuenta admin accesible (`paz.fgabriel@gmail.com` para Wave A; deuda tecnica post-MVP migrar a `admin@nuestrascuentitas.com`)
- Resend API key + DKIM/SPF validados en DNS
- Acceso SSH al VPS turismo

## Pasos
1. Crear proyecto en Dokploy `idp-zitadel-teslita`
2. Levantar Postgres dedicada (Dokploy > Application > Postgres > PostgreSQL v16) con volume persistente. Guardar credenciales en Infisical.
3. Crear Application > Docker > ghcr.io/zitadel/zitadel:<VERSION pinned>
4. Aplicar env vars del `infra/dokploy/zitadel/env.example` (valores de Infisical)
5. Crear dominio Dokploy `id.nuestrascuentitas.com` con cert Let's Encrypt
6. Deploy Postgres primero, esperar healthcheck
7. Deploy Zitadel. Esperar a que `GET /debug/healthz` devuelva 200
8. Crear admin initial (vía env vars o primer login de consola)
9. Seguir pasos T2.1-T2.5 del plan Wave A (Orgs, SMTP, admin migration, OAuth Clients, policies)
10. Deploy backup cron (T3.1)
11. Smoke test completo (T4.1)

## Validacion
- `curl https://id.nuestrascuentitas.com/.well-known/openid-configuration` -> 200 JSON
- `mkey pull bitacora prod --stdout | grep -c '^ZITADEL_'` -> >= 25
- Backup cron ejecuto al menos 1 vez

## Rollback
- Si bootstrap falla a mitad: `dkp.sh POST application.stop <zitadelAppId>` + `dkp.sh POST application.stop <postgresAppId>` + eliminar volume
- Si el masterkey se pierde en el medio: regenerar nuevo masterkey, **descartar DB** (encryption imposible sin masterkey original)

## Escalacion
- Rate-limit Let's Encrypt: usar LE staging (`--caserver https://acme-staging-v02.api.letsencrypt.org/directory`) y no volver a prod hasta root cause
- DKIM/SPF no pasan: NO cerrar Wave A hasta resolver
- OOM Zitadel en primer boot: subir a VPS dedicado o ajustar `ZITADEL_WEBSOCKET_CONNECTIONS`
```

### `infra/runbooks/zitadel-recovery.md` (skeleton)
```md
# Zitadel Teslita — Recovery Runbook

## Escenarios cubiertos
1. Zitadel container down (DB intacta)
2. Postgres container down (volume intacto)
3. Volume Postgres corrupto (requiere restore backup)
4. Masterkey perdido (destroy-and-recreate total)
5. Admin locked out (sin MFA recovery codes)
6. Cert TLS expirado

## Escenario 1: Zitadel container down
- `dkp.sh status <appId>` -> estado deployment
- `dkp.sh logs <appId>` -> error detallado
- Accion comun: `dkp.sh redeploy <appId>` + ver healthcheck

## Escenario 2: Postgres container down
- idem, pero Zitadel mostrara 503 hasta que Postgres vuelva
- Si volume intacto: restart simple resuelve
- Si volume corrupto: ver escenario 3

## Escenario 3: Volume corrupto
- Crear volume nuevo `zitadel-postgres-data-recovery-<ts>`
- Restaurar backup mas reciente: ver `zitadel-backup.md` seccion restore
- Apuntar postgres container al volume nuevo
- Validar login admin + org list

## Escenario 4: Masterkey perdido
- **NO regenerar masterkey**: destruye encryption de DB
- Si no se puede recuperar: destroy-and-recreate (zitadel-bootstrap.md from scratch)
- Mitigacion: masterkey siempre en Infisical (T3.2)

## Escenario 5: Admin locked out
- Si hay TOTP recovery codes en Infisical (T2.3): usar uno en `ui/login`
- Si no: `mkey pull bitacora prod --stdout | grep ZITADEL_ADMIN_` y reintentar
- Si perdieron todo: `docker exec <pg-container> psql -d zitadel -c "UPDATE users SET ..."` (rutas en docs oficiales Zitadel troubleshooting)

## Escenario 6: Cert TLS expirado
- Let's Encrypt autorenueva. Si falla: revisar Traefik logs (`dkp.sh logs <appId>` del Traefik) y CAA records DNS
- Si rate-limit: esperar 7d o usar LE staging env hasta resolver root cause
```

### `infra/runbooks/zitadel-backup.md` (skeleton)
```md
# Zitadel Teslita — Backup Runbook

## Como funciona
- Script: `/opt/zitadel-backup/snapshot.sh` en VPS turismo
- Cron: 03:00 UTC diario (`/etc/cron.d/zitadel-backup`)
- Proceso: stop postgres container -> tar del volume -> rclone upload offsite -> start container -> cleanup retention
- Local retention: 30d en `/var/backups/zitadel/`
- Offsite retention: 90d en `${ZITADEL_BACKUP_OFFSITE_BUCKET}`
- Downtime por backup: ~1-2 min (aceptable a las 03:00 UTC)

## Como ejecutar manualmente
```bash
ssh turismo
/opt/zitadel-backup/snapshot.sh
```

## Como verificar que corrio
```bash
ssh turismo "ls -la /var/backups/zitadel/ | tail -5"
# Y offsite:
rclone ls <provider>:<bucket>/zitadel-pg- | sort -r | head -5
```

## Como restaurar
1. Descargar backup: `rclone copy <provider>:<bucket>/zitadel-pg-<ts>.tar.gz /tmp/`
2. Stop current Postgres: `dkp.sh POST application.stop <postgresAppId>`
3. Crear volume nuevo: `docker volume create zitadel-pg-restore-<ts>`
4. Extraer tar: `docker run --rm -v zitadel-pg-restore-<ts>:/data -v /tmp:/backup busybox tar -xzf /backup/zitadel-pg-<ts>.tar.gz -C /`
5. Reapuntar Postgres a volume nuevo (via Dokploy UI o compose)
6. Start Postgres + verificar healthcheck
7. Start Zitadel + verificar login admin

## Como rotar destino offsite
- Setear nuevas credenciales en Infisical
- `rclone config` interactively en VPS
- Regenerar `/opt/zitadel-backup/.env`
- Ejecutar un backup manual para validar

## Alerta: backup no corrio 2 dias
- Revisar log: `tail -50 /var/log/zitadel-backup.log`
- Causas comunes: rclone creds expiraron, espacio disco agotado, cron deshabilitado
- SEV: alta — IdP sin backup >72h es riesgo operacional
```
