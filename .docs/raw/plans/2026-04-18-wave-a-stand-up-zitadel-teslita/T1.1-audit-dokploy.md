# Task T1.1: Audit Dokploy apps + env + volumes (Zitadel stack)

## Shared Context
**Goal:** descubrir las apps Dokploy del stack Zitadel (frontend + postgres), sus env vars (masked), volumes, domains, estado Traefik.
**Stack:** Dokploy REST API via `dkp.sh`, SSH al VPS turismo (si sshr disponible) o docker commands via dokploy.
**Architecture:** Zitadel corre detras de Traefik con Let's Encrypt en el VPS turismo. Hay que identificar exactamente que apps + volumes forman el stack para poder retro-generar compose en T3.3.

## Locked Decisions
- Usar `dkp.sh` (POSIX wrapper) con DOKPLOY_API_KEY desde `infra/.env` (seteado en T0.1).
- Mask todos los valores de env vars que luzcan como secrets (match regex `(?i)(KEY|SECRET|PASSWORD|TOKEN|PAT|MASTERKEY)` -> reemplazar con `***`).
- Evidencia se guarda en `.docs/raw/reports/2026-04-18-wave-a-audit/T1.1-dokploy-audit.md`.
- No mutar nada. Read-only.

## Prompt
1. Listar proyectos Dokploy: `dkp.sh projects > /tmp/dokploy-projects.json`. Parsear con `jq` para identificar cual proyecto contiene "zitadel" o similar. Si no hay match literal, listar **todos** los `applicationId` de cada proyecto y buscar en el siguiente paso.
2. Por cada candidato:
   - `dkp.sh status <appId> > /tmp/app-<appId>.json` — trae appName, dominio, env, buildArgs, source (git/docker), healthCheck.
   - Identificar appName real (no slug) y qye apuntaesa `domains[].host`.
3. Filtrar las 2 apps que matchean:
   - **Zitadel webapp:** la que tiene domain `id.nuestrascuentitas.com`.
   - **Zitadel Postgres:** la que mencione `postgres` en appName, y cuyas env vars incluyan `POSTGRES_*` relacionados con Zitadel (no confundir con `postgres-reboot-solid-state-application-l55mww` que es la de Bitacora).
4. Si no hay una Postgres dedicada separada de la de Bitacora: **STOP y reportar al dispatcher como critical escalation**. Wave A no puede continuar sin aislamiento de Postgres (decision lockeada).
5. Recolectar los siguientes campos en el reporte:
   - **Zitadel webapp:** appId, appName, dominio, `certificateType`, source (image tag o repo+commit), env vars (nombres + valores masked), buildArgs, volumes mounted, Traefik router labels, startup command, deployments history ultima semana.
   - **Zitadel Postgres:** appId, appName, volumes (**nombre exacto del volumen docker**, critico para T3.1), env vars masked (usuario DB, database name, puerto interno), network, backups config (si existe en Dokploy).
6. Extraer del env de la webapp las keys que deberian persistirse en Infisical en T3.2 (masked en el reporte, valores reales NO incluir):
   - `ZITADEL_MASTERKEY` (critico)
   - `ZITADEL_DATABASE_POSTGRES_*` (host, port, user, pass, database)
   - `ZITADEL_EXTERNALDOMAIN`
   - `ZITADEL_EXTERNALSECURE`
   - `ZITADEL_EXTERNALPORT`
   - `ZITADEL_TLS_*` (si externo)
   - `ZITADEL_SMTP_*` (si ya wireado)
   - `ZITADEL_FIRSTINSTANCE_*` (admin bootstrap)
7. Validar version: `curl -sS https://id.nuestrascuentitas.com/debug/healthz` y/o inspeccionar el image tag en Dokploy (ej: `ghcr.io/zitadel/zitadel:v2.58.3`). **Si la tag es `:latest`, flaggear como drift (decision lockeada: pinnear version)**. Reportar version exacta.
8. Escribir reporte con la estructura del skeleton abajo.

## Execution Procedure
1. `bash $HOME/.claude/skills/dokploy-cli/scripts/dkp.sh projects` -> buscar proyecto con "zitadel" o "idp".
2. Si no matchea, recorrer todos los proyectos con `.services.applications[]` y `.services.postgres[]` buscando dominio `id.nuestrascuentitas.com` o image `zitadel/zitadel`.
3. Para cada app candidata, `dkp.sh status <appId>` y capturar JSON.
4. Usar `jq` + `sed` para masking de values: `jq -r '.env' | sed -E 's/(KEY|SECRET|PASSWORD|TOKEN|PAT|MASTERKEY)=[^\n]*/\1=***/Ig'`.
5. Persistir el reporte en `.docs/raw/reports/2026-04-18-wave-a-audit/T1.1-dokploy-audit.md`.
6. Commit: `docs(report): T1.1 dokploy audit of zitadel stack`.
7. Si algun paso falla con `Unauthorized`: stop y reportar que la DOKPLOY_API_KEY no tiene permisos suficientes; pedir al user una key con scope full.
