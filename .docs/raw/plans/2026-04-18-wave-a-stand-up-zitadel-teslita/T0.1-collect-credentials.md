## Prompt
**Este task requiere interaccion con el user. No ejecutar agent sin que el user provea los inputs.**

Ejecutar los 4 sub-steps **en orden estricto**. Cada sub-step tiene una validacion explicita. Si falla, parar y reportar al user — no improvisar.

### Sub-step 1 — DOKPLOY_API_KEY
1. Pedir al user: "En Dokploy UI, ir a Settings > Profile > API Keys, generar una nueva key con scope `full` (o equivalente read+write), copiarla y pegarla aqui". Tambien pedir confirmacion de la URL Dokploy (default `http://dokploy.gestionturismo.xyz:3000`).
2. El user pega la key via AskUserQuestion opcion "Other" (free-form) o la setea via una linea shell ad-hoc. NO logearla en el output.
3. Agregar al final de `infra/.env`:
   ```
   DOKPLOY_API_KEY=<valor pegado>
   DOKPLOY_URL=http://dokploy.gestionturismo.xyz:3000
   ```
4. Validar: `bash $HOME/.claude/skills/dokploy-cli/scripts/dkp.sh projects` debe devolver JSON `[{"projectId":...}]`. Si devuelve `Unauthorized`, pedir al user que valide la key (puede estar con scope limitado).
5. Persistir en Infisical: `bash $HOME/.claude/skills/mi-key-cli/scripts/mkey.sh set bitacora DOKPLOY_API_KEY "<valor>" --env prod` + `mkey.sh set bitacora DOKPLOY_URL "http://dokploy.gestionturismo.xyz:3000" --env prod`. Validar que `mkey pull bitacora prod --stdout | grep DOKPLOY_API_KEY` imprime la key despues.
6. **Nunca commit `infra/.env`** — el archivo es gitignored.

### Sub-step 2 — Infisical CLI login
1. Correr `bash $HOME/.claude/skills/mi-key-cli/scripts/mkey.sh status`. Si `teslita` dice `[INFO] Sin token env/file visible. Puede existir una sesion CLI en keyring`, probar `mkey pull bitacora prod --stdout > /dev/null 2>&1`. Si falla con `InfisicalProfileSwitchFailed`, re-login requerido.
2. Pedir al user: "Voy a correr `infisical login --domain=http://100.115.71.27:9881`. Eso abre un browser para que pegues tu email + OTP. Confirmame cuando hayas logueado".
3. Correr: `bash $HOME/.claude/skills/mi-key-cli/scripts/mkey.sh login teslita`. Esperar al user.
4. Validar: `mkey pull bitacora prod --stdout | awk -F= 'NF>=2 {print $1}' | sort -u | head -20` debe imprimir lista de keys sin error. Contar lineas debe ser >= 5.
5. Si en vez de re-login el user prefiere Machine Identity UA (recomendado multi-project): pedirle `INFISICAL_TESLITA_UA_CLIENT_ID` + `INFISICAL_TESLITA_UA_CLIENT_SECRET` y setearlos en env User-scope o Machine-scope via PowerShell (`[Environment]::SetEnvironmentVariable('INFISICAL_TESLITA_UA_CLIENT_ID','...','User')`). Validar con `mkey status` que pase a `[ENV]`.

### Sub-step 3 — Zitadel admin access
1. Pedir al user: "Necesito credenciales admin del Zitadel en `id.nuestrascuentitas.com`. Tres opciones: (a) email+password del primer admin que creaste cuando deployaste, (b) Personal Access Token (PAT) emitido en Zitadel UI > Service Users > Personal Access Tokens, (c) abrir una sesion admin en el browser y pasarme un cookie de sesion (`zitadel.session` o similar). Cual usamos?"
2. Guia para (a): correr `curl -sS -X POST https://id.nuestrascuentitas.com/oauth/v2/token -d 'grant_type=password&username=<email>&password=<pass>&scope=openid urn:zitadel:iam:org:project:id:zitadel:aud' -H "Authorization: Basic <client:secret del IAM admin default client>"` — NO ejecutarlo aqui, solo testear que las creds funcionan logueando el user en el browser https://id.nuestrascuentitas.com/ui/login.
3. Guia para (b): al user crear PAT con scopes `urn:zitadel:iam:org:write`, `urn:zitadel:iam:user:write`, `urn:zitadel:iam:org:project:write`, `urn:zitadel:iam:org:project:grant:write`, `urn:zitadel:iam:org:project:app:write`. Guardar el PAT en Infisical: `mkey set bitacora ZITADEL_ADMIN_PAT "<pat>" --env prod`. Validar con `curl -sS -H "Authorization: Bearer $(mkey get ZITADEL_ADMIN_PAT)" https://id.nuestrascuentitas.com/management/v1/orgs/me` retorna JSON con la org default.
4. Si el user no recuerda las creds y no puede loguear: **STOP y escalar**. No proceder sin admin. Opciones: (i) el user busca en su password manager; (ii) resetea pass via DB directo (requiere masterkey para no romper encryption — esta en Dokploy env probablemente); (iii) destroy-and-recreate (Wave A salta a estrategia opcion b — escalar y repnalar).
5. Done cuando: el user confirma que `ZITADEL_ADMIN_PAT` esta en Infisical y `curl -H "Authorization: Bearer $PAT" https://id.nuestrascuentitas.com/management/v1/orgs/me` devuelve 200.

### Sub-step 4 — Mailbox admin@nuestrascuentitas.com
1. Pedir al user: "Necesitamos mailbox `admin@nuestrascuentitas.com`. Opciones: (a) Zoho Mail free (1 user, dominio custom, requiere MX DNS records), (b) Fastmail trial 30 dias (ddns simples), (c) Google Workspace trial ($6/user/mes), (d) seguir con `buhosoft70@gmail.com` transitorio y migrar post-MVP. Cual elegimos?"
2. Si (a-b-c): el user se registra al proveedor y configura MX records en el DNS de `nuestrascuentitas.com` apuntando al proveedor. El user confirma cuando el mailbox pueda recibir mails (test: enviar manualmente a admin@nuestrascuentitas.com desde buhosoft70@gmail.com, y confirmar que llego).
3. Si (d): **reabrir decision de brainstorming** — la decision lockeada fue "admin@nuestrascuentitas.com ahora". Escalar al user para confirmar fallback transitorio y documentar en issue.
4. Done cuando: el user confirma que la dir `admin@nuestrascuentitas.com` recibe mails o que quedamos con `buhosoft70@gmail.com` transitorio documentado.
