## Bloqueador: PATs no extraibles

Los PATs se generaron pero el valor del token no es recuperable. Esto bloquea toda Wave 2 (T2.1-T2.5) que requiere llamar `/management/v1/*` y `/admin/v1/*` con Authorization Bearer.

**Rutas para desbloquear (requieren decision del user):**

1. **Re-bootstrap con volume bind mount para capturar PAT**
   - Stop Zitadel + drop DB `shared_zitadel_v4` + add volume bind `/var/zitadel-bootstrap` al container + change `ZITADEL_FIRSTINSTANCE_PATPATH=/bootstrap/admin-sa.pat` → el bind se mapea a host. Necesita otra forma de leer el file (otra app con mismo mount, o SSH host).
   - Costo: ~15-20 min, requiere configurar volumen compartido.

2. **Deploy login companion app (`zitadel-login:v4.9.0`) + login humano**
   - Nueva App Docker en Dokploy con env `ZITADEL_API_URL=https://id.nuestrascuentitas.com`. Pero companion TAMBIEN necesita `ZITADEL_SERVICE_USER_TOKEN` (el PAT de `login-client`) — que no tenemos. Chicken-and-egg.
   - Excepcion: si `zitadel-login` corre sin service user (modo "unauth"), se puede usar para login humano admin → crear PAT via UI desde Service Users.
   - Costo: ~30 min, requires verificar behavior companion v4.9.0 sin service user token.

3. **SSH VPS turismo + docker exec para extraer PAT runtime**
   - Requiere acceso SSH root al VPS 54.37.157.93 del user (no disponible en mi sesion).
   - Comando: `docker exec $(docker ps -qf name=app-program-optical-matrix) sh -c 'zitadel admin pat ...'` (hipotetico — comando CLI Zitadel puede no existir).
   - Alternativa mas simple: `docker logs $(docker ps -qf name=app-program-optical-matrix) 2>&1 | head -100 | grep -iE "pat|token"` para ver si el PAT esta en logs pre-capture.

4. **DB surgery: INSERT nuevo user + PAT via manipulacion de eventstore**
   - Muy fragil. Zitadel event-sourced necesita el evento exacto con firma/crypto correcta.
   - Descartado.
